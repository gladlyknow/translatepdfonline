# 预览加载与分片优化

## 现状与目标

- **现状**：预览时左侧加载整份源 PDF（R2），右侧加载整份译文（经后端代理），几十 MB 导致十几秒等待。
- **目标**：
  - 单页/多页翻译：源文件只加载任务涉及页（存 R2 切片）；译文在已有“仅含翻译页”的基础上，大文件通过 Range 按需拉取。
  - 整份翻译：译文大文件同样通过 **Range 分片加载**（译文过大问题按 Range 切分即可，不一次拉全）。
  - 翻译进行中增加提示：“因翻译多页/加载 AI 大模型需要一点时间，请多点儿耐心”。

## 架构要点

```mermaid
flowchart LR
  subgraph worker [Worker]
    BabelDOC[BabelDOC translate]
    Extract[PyMuPDF 按页范围提取]
    R2Up[上传 R2]
    BabelDOC --> R2Up
    Extract --> R2Up
  end
  subgraph backend [Backend API]
    File[/tasks/id/file]
    Source[/tasks/id/source-file]
    File --> R2Stream[R2 stream + Range]
    Source --> R2Stream
  end
  subgraph frontend [Frontend]
    Left[左栏 源]
    Right[右栏 译文]
    Left --> Source
    Right --> File
  end
```

- 有 `page_range` 的任务：Worker 用 **PyMuPDF** 从整份 PDF 中按页范围提取，生成 **source_pages.pdf** 上传 R2；预览时左侧只请求该切片（小文件）。
- 译文：继续单文件产出；**译文过大时通过 Range 切分加载**，后端对 `/file` 支持 Range，前端/PDF.js 按需请求字节范围。

---

## 1. 源文件“按任务页”切片并存入 R2

**目的**：有 `page_range` 时，预览左侧只加载该任务涉及的页，而不是整份几十 MB 的源文件。

- **Worker**（[backend/app/tasks_translate.py](backend/app/tasks_translate.py)）
  - 有 `task.page_range` 时：用 **PyMuPDF** 从本地源 PDF `local_path` 按 `page_range` 提取页，生成临时 **source_pages.pdf**，上传 R2：`translations/{task_id}/source_pages.pdf`，并写入 `task.source_slice_object_key`。
  - 无 `page_range` 时不做切片。

- **数据模型**（[backend/app/models.py](backend/app/models.py)）
  - `TranslationTask` 新增可空字段：`source_slice_object_key`（如 `String(512)`），并做 Alembic 迁移。

- **TaskView 与预览 URL**（[backend/app/routes/tasks.py](backend/app/routes/tasks.py)）
  - 若 `task.source_slice_object_key` 存在：`source_pdf_url = f"/api/tasks/{task_id}/source-file"`；否则保持整份文档 R2 公网 URL。

- **新接口：GET /api/tasks/{task_id}/source-file**
  - 若有 `source_slice_object_key` 则从 R2 流式返回该 object；可复用 R2 流式逻辑并支持 Range（见下节）。

- **前端**：左侧继续用 `taskView.source_pdf_url`（有切片时已是 source-file 地址）。若 PDF 经 Next 代理，需新增 **GET /api/tasks/[taskId]/source-file** 的 Route 并转发 Range。

---

## 2. 译文（及源）接口支持 Range，实现分片加载

**目的**：**译文过大问题按 Range 切分**——大 PDF 不一次拉全，由 PDF.js 按需发 Range 请求，后端从 R2 按范围返回 206。

- **R2 层**（[backend/app/storage_r2.py](backend/app/storage_r2.py)）
  - 支持按 `Range` 请求（如 boto3 `get_object(..., Range='bytes=0-65535')`），返回便于 206 的元数据（content_type、stream、content_length、content_range）。

- **GET /api/tasks/{task_id}/file** 与 **GET /api/tasks/{task_id}/source-file**
  - 读取 `request.headers.get("Range")`，若合法则用 R2 Range 取流，响应 **206 Partial Content**，设置 `Content-Range`、`Accept-Ranges: bytes`、`Content-Length`（范围长度）；无 Range 时保持 200 全量流。

- **Next.js 代理**（[frontend/app/api/tasks/[taskId]/file/route.ts](frontend/app/api/tasks/[taskId]/file/route.ts) 及新增 source-file route）
  - 转发 **Range** 头，并透传 206 与 `Content-Range` 等头。前端 PDF 组件无需改，react-pdf/pdf.js 会自动发 Range 请求。

## 3. 多页/整份翻译时的“耐心”提示

- 在 [frontend/messages/zh.json](frontend/messages/zh.json)、[frontend/messages/en.json](frontend/messages/en.json) 等增加文案（如 `translationPatience`）。
- 在 [frontend/app/[locale]/page.tsx](frontend/app/[locale]/page.tsx) 中，当 `taskStatus === "queued" || taskStatus === "processing"` 时在进度区域展示该提示。

---

## 4. 可选（后续）：译文按“多段文件”切分

若 Range 方案在部分环境下仍不理想，可再考虑 Worker 将译文按页/按大小切成多个 part 上传、后端按 part 返回。本次以 **Range 切分译文** 为主。

## 5. 依赖与注意事项

- **PyMuPDF**：**项目尚未使用，可新增依赖 pymupdf**，用于从整份 PDF 中按页范围提取并生成 **source_pages.pdf**（如 `pip install pymupdf`，在 backend 的 requirements 或 pyproject 中加入）。
- **迁移**：新增 `source_slice_object_key` 后执行 Alembic 迁移。
- **R2**：确认 boto3 `get_object` 的 `Range` 参数与 206 响应行为，按文档正确设置并解析响应头。
