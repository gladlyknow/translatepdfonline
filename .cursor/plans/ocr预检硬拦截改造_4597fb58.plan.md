# OCR 真实页数预检方案

## 目标
创建前预检必须使用真实 `documents.pageCount`，并保证该值在上传后及时可用，避免误报：
- 上传后点击 OCR 不应再出现 `document_pages_required_for_billing` 误伤
- 低余额用户（如 30 积分）对超大页数文件必须在创建前被 402 拦截
- 保持完成后仍按 OCR 真实识别页数扣费（现有逻辑）

## 改造范围
- [`frontend/src/app/api/ocr/tasks/route.ts`](d:/imppro/translatepdfonline/frontend/src/app/api/ocr/tasks/route.ts)
- 上传完成相关 API（`/api/upload/presigned/complete` 或其共享落库逻辑）
- 文档查询 API（必要时补充 pageCount 就绪态字段）
- （可选）新增页面数提取/补偿任务工具模块

## 实施步骤
1. 上传完成后强制回填真实页数（主路径）
   - 在上传完成链路解析 PDF 实际页数并写入 `documents.pageCount`。
   - 将 `pageCount` 写入和 `document` 可用状态尽量放在同一事务/同一成功响应语义内，减少“文档已可见但页数未就绪”窗口。
   - 若解析失败，明确标记 `pageCount` 未就绪原因（便于后续补偿）。

2. OCR 创建前预检仅使用真实页数
   - 保留 `doc.pageCount` 校验与积分计算口径：`creditsEstimated = pageCount * creditsPerPage`。
   - 若 `pageCount` 缺失，不立即误伤为失败；改为“短时等待 + 快速重读文档 + 再判定”机制（例如数百毫秒~数秒内有限重试）。
   - 重试后仍缺失才返回明确错误码（可继续使用 `document_pages_required_for_billing`，但文案改为“页数解析中，请稍后重试”）。

3. 消除“上传后立即点 OCR”误伤窗口
   - 在前端 OCR 启动入口读取 `document.page_count`，未就绪时临时禁用提交按钮并提示“正在解析页数”。
   - 与后端短时等待机制叠加，前后端双保险避免误伤。

4. 统一错误码与可观测性
   - 记录 `page_count_value`、`page_count_ready`、`page_count_source`、`page_count_fill_latency_ms`。
   - 记录 OCR 创建预检拦截原因：`insufficient_credits` / `page_count_not_ready`。

5. 保持完成后实际扣费逻辑不变
   - 继续沿用已改造的“按 `ocr-parse-result.json` 的 `pages.length` 结算”路径。
   - 创建前 `creditsEstimated` 仅用于余额预检与展示，不作为最终扣费依据。

## 验证清单
- 上传后立即点击 OCR：不再出现误伤性 `document_pages_required_for_billing`。
- 30 积分 + 高页数文件：创建阶段 402 拦截（基于真实页数）。
- 余额充足且页数就绪：可正常创建并入队。
- 任务完成后：账本扣费与 OCR 识别页数一致，且与 export 点击无关。
