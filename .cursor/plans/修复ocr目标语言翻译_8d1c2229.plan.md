# OCR 解析与翻译全量移植计划

## 目标与范围
- 目标是将 [`D:/imppro/onlinepdftranslator/src/shared/lib/translator`](D:/imppro/onlinepdftranslator/src/shared/lib/translator) 的 OCR 解析与 DeepSeek 翻译链路完整迁移到 `translatepdfonline`，而不是只做局部补丁。
- 行为要求：
  - 已选目标语言且与源语言不同：必须翻译 Markdown 与 parse-result JSON。
  - 未选目标语言或同语种：严格只 OCR，不触发翻译。
- 可靠性要求：
  - OCR-only 与 OCR+translate 全流程都支持失败后重试。
  - 重试必须从断点阶段继续，不允许总是从头开始。
- 输出一致性要求：Workbench 展示、`/api/tasks/:id/parse-result`、md/pdf 导出三者语言一致。

## 分阶段实施
- 阶段 1：建立移植边界与文件映射
  - 源模块分组（来自 onlinepdftranslator）：
    - 核心翻译：`deepseek-translate.ts`、`types.ts`、`pool.ts`、`json-to-markdown.ts`
    - OCR 解析：`baidu-ocr-result.ts`、`baidu-ocr.ts`、`secrets.ts`
    - parse-result 生态：`parse-result-document.ts`、`image-proxy.ts`、`parse-result-export-md.ts`、`parse-result-export-html.ts`
    - 流水线与观测：`pipeline.ts`、`step-runner.ts`、`work-log.ts`、`public-error.ts`
  - 目标承接点：
    - [`D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-translate.ts`](D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-translate.ts)
    - [`D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-queue.ts`](D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-queue.ts)
    - [`D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-export-queue.ts`](D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-export-queue.ts)
    - [`D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-parse-result-target-translate.ts`](D:/imppro/translatepdfonline/frontend/src/shared/lib/ocr-parse-result-target-translate.ts)

- 阶段 2：核心解析与翻译全量迁移
  - 以 onlinepdftranslator 的 `deepseek-translate.ts` 为基线完整迁移：
    - 分段策略、表格策略、覆盖率统计、回退规则、重试策略全部保留。
    - 保持源数据结构不变，不改变现有数据库字段。
  - 将 `baidu-ocr-result.ts` 中 parse 结构抽取逻辑迁入当前 OCR parse 落盘路径，减少“字段没被翻译到”的遗漏。

- 阶段 3：parse-result 全链路接入
  - 统一 source/target JSON 键与读取规则（target 优先，source 回退）。
  - Workbench API、导出队列、markdown 导出全部改为同一读取入口，避免不同页面读取不同语言版本。
  - PATCH 保存逻辑按当前任务语言状态自动落盘到正确 key。

- 阶段 4：队列与流水线语义对齐
  - 把 `pipeline.ts` + `step-runner.ts` 的关键阶段日志与错误分层语义并入现有 `ocr-queue.ts`。
  - 明确每阶段可重试错误与终止错误，避免 silent fallback 造成“看起来成功但实际未翻译”。

- 阶段 4.5：断点续跑状态机与重试能力全量移植
  - 将 onlinepdftranslator 的步骤执行语义移植为任务状态机：
    - 推荐阶段：`ocr_submit_poll` -> `ocr_parse_persisted` -> `translate_markdown` -> `translate_parse_result` -> `export_outputs` -> `completed`
    - OCR-only 路径跳过翻译阶段直接进入 `export_outputs`。
  - 断点持久化：
    - 每阶段完成后持久化 `progressStage`、`progressPercent`、阶段输出 key、尝试次数、最后错误。
    - 对已完成阶段产物（如 source parse/source md/translated md/target parse）做幂等存在性校验，避免重复写入。
  - 失败重试规则：
    - transient error：仅重试当前阶段，不回滚已完成阶段。
    - terminal error：任务标记 failed，用户点击重试时从 `progressStage` 继续。
    - cancel 后恢复：从最近未完成阶段继续，保留已落盘中间产物。
  - 队列消费要求：
    - 同一任务单飞（lease + 幂等 claim）避免并发重复推进阶段。
    - 队列失败回退 dispatcher 时仍保持阶段断点信息不丢失。

- 阶段 5：回归验证与灰度
  - en -> zh：验证 target JSON 生成、Workbench 中文、导出中文。
  - en -> 空：验证仅 OCR，不调用 DeepSeek。
  - en -> en：验证同语种不翻译。
  - 失败重试场景：
    - OCR 成功后翻译失败：重试应从翻译阶段继续。
    - Markdown 翻译成功但 parse-result 翻译失败：重试应从 parse-result 翻译阶段继续。
    - 导出失败：重试应只重跑导出阶段。
    - 任务 cancel 后 retry：应从最后断点继续，不重跑已完成阶段。
  - 追加异常场景：DeepSeek 超时/返回结构异常/R2 读写失败，验证日志、错误分类和用户可见提示。

## 验证方案
- 接口层：
  - `POST /api/ocr/tasks` 入参 `target_lang` 空值时，存储后应等于 `source_lang`。
  - `GET /api/tasks/:id/parse-result` 在异语种任务优先返回 target JSON。
  - `POST /api/ocr/tasks/:taskId/retry` 返回的恢复阶段应与失败断点一致。
- 数据层：
  - 检查 `translations/{taskId}/ocr-parse-result.json` 与 `ocr-parse-result-target.json` 生成时机正确。
  - 检查任务失败后重试不覆盖已完成阶段产物（对象 key 与内容稳定）。
- UI 层：
  - OCR workbench 展示语言与任务目标语言一致。
  - md/pdf 下载内容与 workbench 一致。
  - 失败后“重试”触发后，进度从断点阶段前进而非从 0 开始。

## 风险与控制
- 直接全量移植可能引入类型与依赖差异：分阶段迁移并保留编译检查点。
- 旧任务兼容风险：读取侧保留 source 回退，避免历史任务不可读。
- 大文本翻译成本/时延上升：沿用原项目的 batching 与并发上限配置。
- 断点状态漂移风险：以阶段完成事件为唯一提交点，写入数据库与 R2 时采用“先产物后状态提交”的事务化顺序。
- 重试风暴风险：按阶段维度设置最大尝试次数与退避，超过阈值转 terminal error 交由用户手动重试。

## 关键原则
- 以 onlinepdftranslator `translator` 目录为“实现真源”，当前项目以适配接入为主，不做功能降级版重写。
- 翻译触发只由语言判定控制：`languagesNeedTranslation(sourceLang, targetLang)`。
- 不允许“翻译失败但静默返回成功”；必须有可观测日志与明确错误。
- 不允许“重试丢断点”；失败后必须可从最近阶段恢复。
