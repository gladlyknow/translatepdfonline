# FC 长时间无返回：1 次重试 + 回调重试误伤说明

## 1. `_notify_completed_callback_with_retry` 会不会误伤 10 分钟任务？

**不会**（在现有实现语义下）。

- 该逻辑在 [`babeldoc_fc/main.py`](d:\imppro\translatepdfonline\babeldoc_fc\main.py) 里位于：**下载 PDF → `run_translate_local`（可能很长）→ 上传 R2 全部完成之后**，才进入 `_notify_completed_callback_with_retry`。重试的仅是 **对 Next 的 `callback_url` 的短 HTTP POST**（单次 `httpx` 30s 超时 + 间隔 1/2/4/8s，最多 5 次），**不参与** BabelDOC 执行时间。
- 因此「翻译跑了 10 分钟」只拉长 **FC 对 Next 的 HTTP 响应**时间，不会在翻译中途触发「回调重试耗尽 → 422」。

[`reapStaleFcAcceptedTasks`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\invoke-fc.ts) 条件包含 **`progress_stage === 'fc_accepted'`**。该字段是 Next 在 **收到 FC HTTP 200** 之后才写入的（见 [`invoke-fc.ts`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\invoke-fc.ts) `res.ok` 分支）。**FC 仍在执行、尚未返回 200 时**，Next 侧不会出现 `fc_accepted`，因此 **不会被 `reapStale` 误杀**。

唯一需要调参的是 **`TRANSLATE_STALE_FC_ACCEPTED_MINUTES`**：它只作用于「已 fc_accepted、但 callback 未把任务置终态」的僵尸任务，与「FC 单次请求跑了很久」无关。

---

## 2. `dispatch-pending/route.ts` 是否用了队列？

**没有。** 当前是 **带 `x-cron-secret` 的 HTTP Cron** 调用 [`dispatchPendingTranslateFcJobs`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\invoke-fc.ts)：在 DB 里按条件选 `queued` 任务，循环调用 `invokeTranslateFcForTask`。这与 OCR 使用的 **Cloudflare Queues**（[`wrangler.consumer.jsonc`](d:\imppro\translatepdfonline\frontend\wrangler.consumer.jsonc) 里 `OCR_PIPELINE_QUEUE` / `translatepdfonline` 队列 + [`ocr-queue.ts`](d:\imppro\translatepdfonline\frontend\src\shared\lib\ocr-queue.ts)）是两套机制。

---

## 3. 「长时间 queued + FC 没返回」与现状

- **FC HTTP 一直不返回**：Next 侧 `fetch` 带 [`AbortSignal.timeout(FC_FETCH_TIMEOUT_MS)`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\invoke-fc.ts)（默认约 10 分钟，见 [`translate-fc-contract.md`](d:\imppro\translatepdfonline\frontend\docs\translate-fc-contract.md)）。超时进入 `catch`，会写 `fc_next_attempt_at` + 增加 `fc_dispatch_attempt_count`，**由 Cron 反复再调 FC**，次数上**没有上限**（与「只重试 1 次」不符）。
- **积分**：仍在 [`/api/translate/callback`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\callback\route.ts) 成功分支里 `consumeCredits`；只要仍是同一 `task_id`、成功回调一次且幂等字段正常，**重试 FC 不改变「成功后按原逻辑扣费」**（需避免双写：依赖现有 `creditConsumeId` 等幂等逻辑即可）。

## 4. 建议实现：「仅 1 次」自动重试（可选队列）

### 4.1 行为定义（与产品对齐）

- **仅针对**：`invokeTranslateFcForTask` 里 **`fetch` 抛错**（超时、断连、`AbortError` 等）——对应「卡在 FC 没返回 / 连接层失败」。
- **不重试或单独计数**：HTTP 4xx/5xx 里已有 `RETRYABLE_STATUS` 分支（429/502/503/500）若需与「1 次」统一，需在实现时二选一：**要么**只对 `catch` 限 1 次；**要么**全局「自动派发 FC」最多 2 次（首调 + 1 次重试）——需你确认是否包含 429 退避。

### 4.2 实现路径 A（推荐先做，改动小）：纯 DB + 现有 Cron

在 `catch` 分支根据 **专用计数或布尔** 判断（避免与 429 退避的 `fcDispatchAttemptCount` 混用建议加列 **`fc_fetch_deadline_retry_used`** `boolean default false`，或复用整数 **`fc_invoke_hang_retry_count`** 上限 1）：

- 若尚未用尽 1 次：置 `fc_next_attempt_at` 为短延迟（或立即）、清 lease、`progress_stage` 可写 `fc_hang_retry_scheduled`，**不**标 `failed`。
- 若已用尽：标 `failed`，`error_code` 如 `fc_invoke_timeout_exhausted`，与现有 `error_message` 一致风格。

Cron **仍用** [`dispatch-pending`](d:\imppro\translatepdfonline\frontend\src\app\api\translate\dispatch-pending\route.ts)，无需新队列即可满足「最多再跑 1 次 FC」。

### 4.3 实现路径 B（你提到的队列）：Cloudflare Queue 投递 1 条

- 新建队列（例如 `translatepdfonline-fc-retry`）及 **producer**（在 `catch` 且仍有 1 次额度时 `send({ task_id })`）。
- **Consumer**（可挂在现有 [`workers/ocr-pipeline-consumer`](d:\imppro\translatepdfonline\frontend\workers\ocr-pipeline-consumer) 同 Worker 增加第二队列 consumer，或单独小 Worker）：收到消息后调用 **`invokeTranslateFcForTask(taskId)`**（需与主应用共享 DB/Hyperdrive 绑定方式，与 OCR consumer 对齐）。
- **防重**：与 Cron 双发同一任务的问题，用 DB 条件更新（仅 `queued`、未 `fc_accepted`、重试位未用尽）+ 短 lease，或「入队后置 `fc_retry_message_id`」避免重复 send。

路径 B 涉及 **wrangler 多文件、队列名、部署与 Dashboard**，工作量明显大于 A；若「队列」仅为解耦与削峰，**路径 A 已满足「重试 1 次」**。
