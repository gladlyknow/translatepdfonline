# PDF 健康检查与 OCR 前置方案

## 一、PDF 健康检查（ToUnicode / 字体子集）

### 1.1 目标与触发时机

- **目标**：在「用户发起页/切片翻译」时，对即将送入 BabelDOC 的 PDF（整份或切片）做轻量检测：ToUnicode 映射是否缺失、字体子集是否明显损坏；若损坏则**不进入 BabelDOC**，直接失败并给用户明确提示。
- **触发时机**：在 Worker 的 `run_translation_task` 中，在**确定 `local_path` 之后、调用 `run_translate()` 之前**执行（与当前“先创建任务再在 Worker 里失败”一致，避免在上传时阻塞且能覆盖切片与整份两种路径）。

### 1.2 检测逻辑（实现位置与方式）

- **新建模块**：`[backend/app/pdf_health.py](backend/app/pdf_health.py)`（或 `backend/scripts/pdf_health_check.py` 再在 tasks 里调用）。不依赖 BabelDOC，仅依赖现有 `pymupdf`。
- **检测项**：
  - **ToUnicode 缺失**：用 PyMuPDF 打开 PDF，遍历所用字体（如 `page.get_fonts()` 或文档级字体列表）；对需要文本提取的字体，判断是否存在 ToUnicode 或可回退的编码信息。若主要文字字体缺少 ToUnicode 且无法用 CMap/Encoding 可靠映射，则判定为「ToUnicode 缺失」。
  - **字体子集损坏**：检查嵌入字体流是否可被正常打开/解析（如 pymupdf 读字体时是否抛异常、或字体流长度/头信息是否明显异常）。若关键字体无法加载则判定为「字体子集损坏」。
- **接口**：例如 `check_pdf_health(path: Path, page_range: str | None) -> tuple[bool, str | None]`，返回 `(True, None)` 或 `(False, "tounicode_missing" | "font_subset_corrupt" | "unknown")`，必要时带简短 detail 字符串供 `error_message` 使用。

### 1.3 与翻译任务衔接

- 在 `[backend/app/tasks_translate.py](backend/app/tasks_translate.py)` 中，在得到 `local_path` 且尚未调用 `run_translate()` 的位置：
  - 调用 `check_pdf_health(local_path, page_range_for_translate)`。
  - 若返回不健康：`_update_task_status(db, task, "failed")`，并设置 `task.error_code = "pdf_font_unsupported"`（或细分 `tounicode_missing` / `font_subset_corrupt`），`task.error_message` 为面向用户的英文/中文说明（如“该 PDF 缺少 ToUnicode 映射或字体损坏，无法可靠提取文字。建议使用带文本层的 PDF 或先进行 OCR 预处理。”），然后 `return`，不调用 BabelDOC。

### 1.4 前端展示

- 任务失败时已有 `[frontend/app/[locale]/page.tsx](frontend/app/[locale]/page.tsx)` 展示 `taskView.task.error_message`，无需改结构。
- 可选：在 `[frontend/messages/zh.json](frontend/messages/zh.json)` / `en.json` 中为 `pdf_font_unsupported`、`tounicode_missing`、`font_subset_corrupt` 增加简短说明，前端根据 `error_code` 优先显示对应文案，否则继续用 `error_message`。

## 二、OCR 前置选项（OCRmyPDF）

### 2.1 产品与架构选择

- **行为**：用户勾选「先用 OCR 处理 PDF」后，翻译流程变为：**取得待译 PDF（整份或切片）→ 调用 OCRmyPDF 生成带文本层的 PDF → 将 OCR 输出作为 BabelDOC 的输入进行翻译**。这样扫描版/图多版 PDF 先被 OCR 成可提取文字，再翻译，提高“可译段落”数量并减少 “no paragraphs” / CID 问题。
- **架构**：采用**翻译请求级可选 OCR**（与市场常见“OCR 预处理 + 再翻译”一致）：
  - 在 `TranslateRequest` 中增加可选字段 `preprocess_with_ocr: bool = False`。
  - 仅当用户勾选时，Worker 在拿到 `local_path` 后先跑 OCR，再对 OCR 输出执行现有 BabelDOC 流程；不勾选则与当前行为完全一致。

### 2.2 后端

- **Schema**：`[backend/app/schemas.py](backend/app/schemas.py)` 的 `TranslateRequest` 增加 `preprocess_with_ocr: Optional[bool] = Field(False, ...)`。
- **Model**：`[backend/app/models.py](backend/app/models.py)` 的 `TranslationTask` 增加 `preprocess_with_ocr: bool = False`（用于记录与展示）；创建任务时从请求体写入。
- **路由**：`[backend/app/routes/tasks.py](backend/app/routes/tasks.py)` 的 `POST /translate` 中读取 `req.preprocess_with_ocr`，写入 `TranslationTask`。
- **Worker**：`[backend/app/tasks_translate.py](backend/app/tasks_translate.py)`：
  - 在得到 `local_path` 且**通过 PDF 健康检查（若实现）之后**，若 `getattr(task, "preprocess_with_ocr", False)`：
    - 生成临时输出路径（如 `output_dir / "ocr_preprocess.pdf"` 或 temp 目录）。
    - 调用 OCRmyPDF：**优先采用 subprocess 调用 `ocrmypdf` CLI**（`ocrmypdf input.pdf output.pdf -l <lang>`），避免将 OCRmyPDF 及其依赖（Python 3.11+、pikepdf 等）强绑进当前 backend 环境；若未安装 CLI 则可回退到尝试 `import ocrmypdf; ocrmypdf.ocr(...)` 并捕获异常。
    - 语言映射：根据 `task.source_lang` 映射到 Tesseract 语言码（如 zh→chi_sim+eng，en→eng，es→spa），传入 `-l`。
    - 成功：将 `local_path` 替换为 OCR 输出路径，后续 `run_translate(local_pdf_path=ocr_output_path, ...)`；失败：设置 `task.error_code = "ocr_preprocess_failed"`，`error_message` 提示安装/配置 OCRmyPDF 与 Tesseract 或重试，然后 `return`。
  - 若未勾选 OCR，逻辑与现在一致，直接 `run_translate(local_path, ...)`。

### 2.3 OCRmyPDF 集成方式建议

- **推荐**：Worker 内通过 **subprocess 调用系统/venv 的 `ocrmypdf` 命令**（如 `ocrmypdf input.pdf output.pdf -l eng --optimize 0`），这样：
  - 不强制 backend 使用 Python 3.11+，也不增加 OCRmyPDF 的依赖到主 requirements.txt。
  - 部署时在 Worker 环境中单独安装 OCRmyPDF（`pip install ocrmypdf` 或系统包）并安装 Tesseract，文档中说明为“可选：启用 OCR 前置时需安装”。
- **备选**：若希望与现有 Python 环境统一，可将 `ocrmypdf` 列为可选依赖（e.g. `pip install ocrmypdf` 在可选组），Worker 内先 `import ocrmypdf`，成功则调用 `ocrmypdf.ocr(input_path, output_path, language=[...])`，失败则提示“未安装 OCRmyPDF”。

### 2.4 前端

- `**[frontend/components/TranslationForm.tsx](frontend/components/TranslationForm.tsx)`**：增加一个复选框（如「先对 PDF 进行 OCR（适用于扫描件或图片较多的 PDF）」），状态 `preprocessWithOcr`，提交时随 `api.translate(..., preprocessWithOcr)` 传入（需在 `[frontend/lib/api.ts](frontend/lib/api.ts)` 的 `translate` 中增加该参数并写入请求体）。
- `**[frontend/messages/*.json](frontend/messages/zh.json)`**：为复选框 label 和说明、以及失败时的 `ocr_preprocess_failed` 增加多语言文案。

## 三、错误码与前端文案统一

- 在 Worker 中为以下情况设置 `error_code` 并给出简短 `error_message`：
  - `pdf_font_unsupported`（或 `tounicode_missing` / `font_subset_corrupt`）：健康检查不通过。
  - `no_paragraphs`：BabelDOC 抛出 "The document contains no paragraphs." 时捕获并设置，提示可能为扫描页或图片页，建议尝试 OCR 前置。
  - `ocr_preprocess_failed`：OCR 步骤失败。
- 前端可根据 `error_code` 显示对应 i18n 文案，并保留回退显示 `error_message`。

## 四、实现顺序建议

1. **PDF 健康检查**：实现 `pdf_health.check_pdf_health()`（PyMuPDF 检测 ToUnicode/字体），在 `run_translation_task` 中调用并处理失败分支；前端沿用现有 error_message 展示，可选补充 error_code 文案。
2. **BabelDOC “no paragraphs” 映射**：在 `tasks_translate.py` 的 except 中识别 "no paragraphs" 异常，设置 `error_code=no_paragraphs` 与友好 `error_message`，并建议尝试 OCR 前置。
3. **OCR 前置**：Schema + Model + 迁移 → Worker 内 subprocess（或 API）调用 OCRmyPDF → 前端复选框与 API 参数；最后补充 OCR 失败时的错误码与文案。

## 五、依赖与部署说明

- **健康检查**：仅用现有 `pymupdf`，无新增依赖。
- **OCR 前置**：需在 Worker 环境中安装 OCRmyPDF 与 Tesseract（可选功能）。若用 subprocess，推荐 `pip install ocrmypdf` 或系统包 + Tesseract；README 中注明“启用 OCR 前置需安装 OCRmyPDF 与 Tesseract”。
