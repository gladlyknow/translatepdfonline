# 删除空态、首页风格与翻译页对比度修复

## 1. 删除当前 PDF 后仍出现「历史/其它文件」

**根因**：[TranslatePageClient.tsx](frontend/src/app/[locale]/(translate)/translate/TranslatePageClient.tsx) 中 `handleDeleteDocument` 在 `deleteDocument` 成功后执行 `listDocuments()`，若 `docs.length > 0` 会 **自动 `setDocumentId(docs[0])`** 并填充 `filename` / `lastUploadedFile`。因此只要账号下还有其它文档，就会立刻进入另一文档的工作台并继续请求 `preview-url`、侧栏仍显示任务历史——这与「删除后应为空」的预期不一致。

**改动**：

- 删除成功后 **不再** 根据 `listDocuments()` 自动选中任何文档；保持 `documentId === null`，清空任务与预览相关 state（现有 `setTaskId` / `setTaskView` 等已做），用户回到 **漏斗空态**（上传区）。
- 若需避免陈旧 `sourceSliceUrl` 等残留，可顺带在删除分支里重置与预览相关的 state（若当前未在删除时清空，可一并检查 `TranslatePageClient` 内 `sourceSliceUrl`、`targetSliceUrl` 等与 `documentId` 绑定的字段）。

服务端 [DELETE /api/documents/[documentId]](frontend/src/app/api/documents/[documentId]/route.ts) 已删除该文档下的 `translation_tasks`，无需改 API。

## 2. 首页风格对齐翻译页（深色 Hero + SaaS 感）

首页由 [page.tsx](frontend/src/app/[locale]/(landing)/page.tsx) + `pages.index` JSON + [themes/default/blocks/hero.tsx](frontend/src/themes/default/blocks/hero.tsx) 渲染。

**改动思路**（优先小侵入、可配置）：

- 在 **Hero 区块 JSON**（如 [en/pages/index.json](frontend/src/config/locale/messages/en/pages/index.json) 的 `sections.hero`，并同步 **zh / es** 若存在同结构）增加可选字段，例如 `variant: "translateDark"`（或 `visual_style`）。
- 在 [hero.tsx](frontend/src/themes/default/blocks/hero.tsx) 中根据该字段分支：
  - 外层 `section`：`bg-gradient-to-b from-slate-950 via-slate-900 to-slate-950`、顶部微弱径向高光（与 [TranslateLandingSections.tsx](frontend/src/shared/components/translate/TranslateLandingSections.tsx) 一致的可复用类名或抽 1 个小组件）。
  - 标题/副标题：`text-zinc-50` / `text-zinc-300`，替代默认 `text-foreground` / `text-muted-foreground`。
  - 公告条、按钮：与翻译页一致的玻璃边框 / 渐变主按钮（可沿用现有 `Button` + `className` 或 `asChild` Link）。
  - **该变体下** 弱化或关闭 `background_image`（避免与深色渐变打架），大图区可保留但加暗色遮罩或缩小占比，避免与翻译页「上图下文」冲突过大。

若 `Section` 类型未包含新字段，在 [shared/types/blocks/landing](frontend/src/shared/types/blocks/landing) 中为 hero 扩展可选 `variant`（或 `Record<string, unknown>` 已足够则仅 JSON + 组件内收窄）。

## 3. 语言选择栏「白底白字」看不见

**根因**：[LanguageSelector.tsx](frontend/src/shared/components/translate/LanguageSelector.tsx) 在 `tone="darkHero"` 时使用 `bg-white/10` + `text-zinc-100`。原生 `<select>` 在 **Windows / Chrome** 等环境下常由系统绘制为 **白底**，浅色字叠在白底上导致不可读。

**改动**：

- `darkHero` 下改为 **不透明或高不透明度深色底**：例如 `bg-slate-900` 或 `bg-slate-800/95`，边框 `border-white/20`，文字 `text-zinc-100`。
- 为 `<select>` 增加 `color-scheme: dark`（Tailwind：`className` 含 `[color-scheme:dark]`），改善部分浏览器下下拉与控件配色。
- （可选）占位 `option` 若仍异常，再评估改为自定义 Listbox（成本较高，通常前两步即可）。

## 4. 顶栏 Pricing / API / History 与背景对比度不足

**根因**：[TranslateShellHeader.tsx](frontend/src/shared/components/translate/TranslateShellHeader.tsx) 在 `appearance === 'onDark'` 时使用 `**bg-white/10` + `backdrop-blur`**。当主内容区向下滚动、浅色区块（三步 / SEO）透过模糊层叠在顶栏后方时，等效背景接近白色，而导航仍为 `**text-zinc-200`**，对比度崩溃。

**改动（推荐先做低成本方案）**：

- 将 `onDark` 顶栏改为 **更高不透明度深色底**，例如 `bg-slate-950/90` 或 `bg-slate-950/85`，保留 `border-white/10`；链接保持 `text-zinc-100` / `hover:bg-white/10`。这样无论下方是深 Hero 还是浅色区，文字都稳定可读。
- **若仍不满意**：在 [TranslateLandingSections.tsx](frontend/src/shared/components/translate/TranslateLandingSections.tsx) 深色段底部放 `id`/`ref` 哨兵，用 `IntersectionObserver` 在漏斗页根据是否滚过 Hero 调用 `setAppearance('onLight'|'onDark')`，并与 `TranslatePageClient` 里「有 `documentId` 强制 `onLight`」的逻辑协调（仅无文档时由滚动驱动）。此方案改动面更大，作为备选。

**SignUser**：若登录按钮在深色顶栏上仍偏淡，可在顶栏 `onDark` 分支给 [SignUser](frontend/src/shared/blocks/sign/sign-user.tsx) 包一层 `className` 或增加可选 `variant`（仅当视觉仍不达标时再做）。

## 验证建议

- 删除当前文档且账号下仍有其它文档：应进入 **无 documentId** 的漏斗页，**不应**再请求其它 `documentId` 的 `preview-url`。
- 首页 Hero：视觉与翻译漏斗深色段一致（渐变、主文案层级、主 CTA）。
- 翻译页：Hero 内语言下拉在浅色/深色系统主题下均可读；滚动过浅色区后顶栏链接仍清晰。
