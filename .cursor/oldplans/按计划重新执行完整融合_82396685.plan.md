2. **需求 2 技能**：技能已移植到 `.cursor/skills/`，路径已改写；但与参考的「页面 JSON 结构 + locale 配置」不一致（参考为 `config/locale/messages/en|zh/pages.`* + `getTranslations('pages.pricing')` 等）。

6. **Phase B 前端融合**：
  - **Auth**：参考为 better-auth + `core/auth` + `app/api/auth/[...all]` + `(auth)/sign-in`、`sign-up`、`verify-email` + `shared/blocks/sign/`*（SignIn、SignUp、SocialProviders）。当前仍为 NextAuth + `app/[locale]/login`、`register`，无 (auth) 路由组、无 verify-email、无 shared sign 区块。

7. **后端**：auth/user 路由已从 main 移除，符合计划；但若前端要「完全」融合，需明确翻译/上传/任务等 API 的鉴权是否仍用现有方式（cookie/匿名）或与参考前端的 session 一致。

### 步骤 7：前端融合注册、登录、支付、订阅、订单（按参考完整移植）

- **Auth**（参考 [core/auth](tmp/onlinepdftranslator/src/core/auth)、[app/api/auth/[...all]](tmp/onlinepdftranslator/src/app/api/auth/[...all]/route.ts)、[shared/blocks/sign](tmp/onlinepdftranslator/src/shared/blocks/sign)）：
  - **方案 A（完整移植）**：在 frontend 引入 better-auth；新建 `frontend/core/auth/`（config + 可选 client）；新建 `frontend/app/api/auth/[...all]/route.ts` 使用 better-auth 的 Next 处理器；新建路由组 `app/[locale]/(auth)/`，下挂 sign-in、sign-up、verify-email 页；从参考移植 `shared/blocks/sign/`* 到 `frontend/shared/blocks/sign/`（SignIn、SignUp、SocialProviders 等），并适配 i18n、Link、authClient。better-auth 需 DB：在 frontend 增加 Drizzle + 与参考一致的 auth 表（user、session、account、verification），使用 `AUTH_DATABASE_URL` 或现有 DB。
  - **方案 B（结构 + UI 先行）**：保留 NextAuth 作为当前鉴权，但：新建 `(auth)` 路由组、`/sign-in`、`/sign-up`、`/verify-email` 路由；从参考移植 sign 区块 UI 到 frontend，并改为调用 NextAuth 的 signIn/signUp 与 session；后续再择机替换为 better-auth。
  - 计划要求「以前端为准」，故需在 frontend 内实现/移植一套完整登录注册流程（含 UI 与 API）；建议采用方案 A 以与参考一致，若 DB 未就绪可先实现方案 B 并预留 better-auth 接入点。
- **User/Config**（参考 [shared/models/user](tmp/onlinepdftranslator/src/shared/models/user.ts)、[shared/models/config](tmp/onlinepdftranslator/src/shared/models/config.ts)）：
  - 若采用 better-auth：`getUserInfo()` 从 session 取 user id 再查 DB user；否则沿用 NextAuth session 提供 user。若需 credits：在 frontend 增加 credit 模型与 `getRemainingCredits`，或先返回 0 占位。
  - Config：参考从 DB 的 config 表读；frontend 可先实现 `getConfigs()` 从 env 或静态 JSON 返回（如 `email_auth_enabled`、`google_client_id` 等），供 auth 与定价页使用。
- **Payment / Order / Subscription**（参考 [app/api/payment](tmp/onlinepdftranslator/src/app/api/payment)、[extensions/payment](tmp/onlinepdftranslator/src/extensions/payment)、[shared/models/order](tmp/onlinepdftranslator/src/shared/models/order.ts)、[shared/services/payment](tmp/onlinepdftranslator/src/shared/services/payment)）：
  - 在 frontend 建立与参考一致的**结构**：`frontend/extensions/payment/`（types、stripe 等 stub 或实现）、`frontend/shared/models/order.ts`、`subscription.ts`、`credit.ts`（含 Drizzle schema 或先接口类型）、`frontend/shared/services/payment.ts`（getPaymentService、handleCheckoutSuccess 等）；`app/api/payment/checkout`、`callback`、`notify/[provider]` 按参考逻辑实现（读 config、getUserInfo、createOrder、paymentProvider.createPayment、updateOrder、redirect）。
  - 若暂不接真实支付：可保留「无支付提供商时返回友好错误」或跳转定价页；订单/订阅表可先建表与 API 形态，写入「待支付」等状态。
- **Settings 与定价页**（参考 [settings/layout](tmp/onlinepdftranslator/src/app/[locale]/(landing)/settings/layout.tsx)、[settings/billing](tmp/onlinepdftranslator/src/app/[locale]/(landing)/settings/billing/page.tsx)、[settings/payments](tmp/onlinepdftranslator/src/app/[locale]/(landing)/settings/payments/page.tsx)、[pricing/page](tmp/onlinepdftranslator/src/app/[locale]/(landing)/pricing/page.tsx)）：
  - 从参考移植 **ConsoleLayout**、**PanelCard**、**TableCard**（或等价）到 frontend（如 `frontend/shared/blocks/console/`、`panel/`、`table/`），并移植 settings 各页：billing（当前订阅 + 订阅列表表）、payments（订单列表表）、credits、invoices、profile、apikeys 等（可按需先做 billing、payments、pricing）。
  - 定价页：参考使用 `getThemePage('dynamic-page')` 且 sections 含 pricing（从 `getTranslations('pages.pricing')` 取）；frontend 的 `/[locale]/pricing` 可改为使用同一套 dynamic-page + pricing 区块，或单独实现定价页但从 messages 或 content 读 pricing 配置，与参考的 `page.sections.pricing` 结构一致。
